# 연구 에이전트 빌드 계획서

# 1\. 개요

본 계획서는 특정 연구 분야에서 반복적이고 시간이 많이 소요되는 작업을 자동화하여 연구 생산성을 극대화하기 위한 '연구 에이전트' 구축의 목표, 범위, 주요 기능 및 개발 일정을 명시합니다. 

## 1.1 희망하는 프로그램 개요

이미지 기반 seq-to-seq 예측 문제를 푸는 모델을 만드는 중임.

* 가볍지만 좋은 성능을 내는 모델을 만들고 싶다.  
* 그 데이터의 도메인의 관점에서 봤을 때 물리적으로, 학술적으로 타당함이 반영된 모델을 만들고 싶다.  
  * 예를 들어 풍력데이터에서는 u와 v의 성질을 반영한 아키텍처 설계라던가, 일사량 데이터에서는 어떤 지역별 특성이라던가 등  
* U-Net 기반으로 연구를 진행하고 싶고, 최신 문헌에서는 어떤 모듈을 가져다 붙여서 성능을 극대화하고 있는지 궁금하다.  
* (현재 단계에서는 진행 X) BT 모형을 통해, 각 논문의 모델들 전부를 실행하지 않고도 비교우위를 갖는 모델을 찾아내고 싶다.

## 1.2 목표

* 문헌 검색 및 요약 과정을 자동화하여 연구 시간 단축  
* 데이터 분석 및 시각화 프로세스 지원  
* 실험 계획 및 결과 보고서 초안 작성 보조

## 1.3 범위

본 프로젝트의 범위는 초기 단계에서 다음 세 가지 핵심 모듈에 초점을 맞춥니다.

* 문헌 search & review draft maker module  
* 문헌 관리 모듈 (Literature Review Module): NAS에 다운업로드 및 업로드 최적화  
* Draft Refiner Module  
* Scorer Module  
* Re-Publisher  
* Professor reviewer module

# 2\. 주요 기능 및 모듈 설계

본 섹션에서는 '연구 에이전트'가 수행할 핵심 기능과 이를 구현하기 위한 주요 모듈의 상세 설계를 제시합니다. 1.3 범위에서 정의된 모듈들을 기반으로, 이미지 기반 seq-to-seq 예측 모델 연구의 생산성을 극대화하는 방향으로 기능이 구체화됩니다.

## 2.1. 문헌 Search & Review Draft Maker Module (문헌 검색 및 초안 생성)

* **주요 기능:**  
  * **최신 문헌 검색:** U-Net 기반의 이미지 seq-to-seq 예측 문제를 다루는 최신 연구 문헌을 자동으로 검색합니다.   
  * **도메인 특화 검색:** 데이터 도메인의 물리적/학술적 타당성을 반영한 아키텍처 설계가 포함된 논문을 우선적으로 필터링합니다.  
  * **초안 생성:** 각 검색된 문헌의 Abstract를 추출해, 연구자가 즉시 확인할 수 있는 문헌 검토(Literature Review) 섹션의 초안을 작성합니다. 각 문헌에 대한 초안에는 아래의 내용이 포함됩니다.  
    1. Abstract 전문: 원문으로 작성  
    2. 해당 문헌을 볼 수 있는 링크: arxiv, (추후 서비스:or open review): 링크는 문헌 관리 모듈에서 사용됩니다.  
    3. Literature Review placeholder(Draft Refiner Module에서 채울 계획)  
    4. Tags(Draft Refiner Module에서 refine할 수도 있음.)  
  * 해당 모듈 한 번의 실행마다 10건의 연구 문헌을 검색하고 각 문헌 별로 1개의 (즉, 총 10개의) 초안을 생성합니다. {paper\_name}의 이름을 가진 블로그 페이지가 하나씩 생성되는겁니다. 완료된 문헌들에 대해서, 해당 모듈이 재가동되었을 때 중복 수집되지 않도록 history.json을 만들고 업데이트 합니다. 

## 2.2. 문헌 관리 모듈 (Literature Review Module)

* **주요 기능:**  
  * **통합 관리:** 검색된 문헌 및 초안 생성 결과물을 체계적으로 저장하고 관리하는 기능을 제공합니다.  
  * **NAS 연동 최적화:** 연구 환경의 NAS(Network Attached Storage) 시스템과의 다운로드 및 업로드 프로세스를 최적화하여, 연구 자료에 대한 접근성과 안정성을 확보합니다.

## 2.3. 초안 개선 모듈 (Draft Refiner Module)

* **주요 기능:**  
  * **Document Taker:** 2.2의 모듈로부터 각 문헌을 순차적으로 전달 받고 아래 작업을 각 문헌에 대해 실시합니다.  
  * **LLM Refiner:** gemini 2.5 pro, gemini 3등의 API를 통해서 아래 질문들에 답을 생성하고, Literature Review placeholder에 채워넣습니다. 이때, 문헌 Search & Review Draft Maker Module에서 작성한 Abstract 원본은 수정하지 않습니다. 2.3 모듈부터는 한글로 글을 작성해야 합니다.  
    1. Abstract을 개조식 요약하고,    
    2. 아래 7가지 각 질문에 대해 각 1\~2 문단 이내로 요약하기  
       1. What is new in the work?  
       2. Why is the work important?  
       3. What is the literature gap?  
       4. How is the gap filled?  
       5. What is achieved with the new method?  
       6. What data are used?  
       7. What are the limitations?  
    3. 사용한 아키텍처에 대한 수식 및 노테이션을 포함한 상세한 설명  
       1. 반드시 포함: loss function, input, output, task, tensor shape…  
    4. Vanilla U-Net과 비교했을 때 추가된 모듈 다시 정리  
    5. Tags 확인 후 추가할 태그 확인  
  * **문체 및 논리 교정:** 2.1 모듈에서 생성된 초안의 문체와 학술적 논리 흐름을 정제하고 개선합니다.  
  * **표현 극대화:** 수식은 엄밀하고 타당해야 합니다. 수식부분은 OCR을 이용해 다시 모델이 수식을 잘 썼는지 확인하도록 설계합니다.

## 2.4. 모델 평가 및 비교 모듈 (Scorer Module)

* **주요 기능 (현재 단계에서는 진행 X):**  
  * **BT 모형 기반 평가:** 2.3 모듈에서 작성한 내용을 기반으로 스코어를 매기는 기준을 만듭니다. 해당 기준을 통해 각 문헌에 대해 score 벡터를 LLM을 통해 매깁니다. BT(Bradley-Terry) 모형 등의 비교우위 분석 모델을 개발하여, 실제 모든 논문의 모델을 실행하지 않고도 각 모델의 상대적인 성능 및 효율성을 예측하고 비교합니다.  
  * **연구 방향성 제시:** 비교 결과를 바탕으로, 연구 목표(가벼우면서도 고성능)를 달성하는 데 가장 유망한 아키텍처나 모듈 조합을 추천합니다.

## 2.5. Re-Publisher (재구조화 모듈)

* **주요 기능:**  
  * **포맷 변환:** 완성된 각 문서에 대한 결과 보고서를  Hugo style 블로그와 호환되도록(이미 **문헌 Search & Review Draft Maker Module**에서 틀을 만듦) 다듬고, publish합니다.

## 2.6. Professor Reviewer Module (교수 검토 지원 모듈)

* **주요 기능:**  
  * **피드백 통합 지원:** 주기적으로 교수님의 자아를 가진 LLM을 설정해, 랜덤으로 3건씩 게시물(각 문헌에 대한 보고서)을 검토하게 합니다. 검토 및 피드백을 기록, 분류하고, 해당 피드백을 문서에 댓글로 남기는 모듈입니다.  
  * **진행 상황 보고서 자동 생성:** 프로젝트 진행 상황 및 주요 결과(문헌 검색, 모델 스코어 등)를 요약하여 정기적인 보고서 초안을 자동으로 생성합니다.

## 2.7. 확장성 및 재사용성 확보 전략 (Scalability & Reusability Strategy)

본 에이전트가 이미지 기반 seq-to-seq 예측 문제뿐만 아니라, **조사 대상 도메인만 변경하여 다른 연구 분야에서도 신속하게 재사용**될 수 있도록 다음과 같은 설계 전략을 적용합니다.

### 2.7.1. 도메인 추상화 레이어 도입

* **목표:** 데이터의 물리적/학술적 타당성(예: 풍력 데이터의 u, v 성질 반영)을 반영하는 로직이나 특화된 검색 조건이 특정 도메인에 종속되지 않도록 분리합니다.  
* **구현:**  
  * 각 도메인(풍력, 일사량, 기타)에 특화된 정보(도메인 특화 아키텍처 키워드, 물리적 제약 조건)를 담은 별도의 **플러그인 파일 또는 Config 설정**으로 관리합니다.  
  * **문헌 Search & Review Draft Maker Module (2.1)** 및 \*\*Draft Refiner Module (2.3)\*\*은 실행 시 이 추상화 레이어에서 도메인 정보를 읽어와 검색 및 정제 작업에 활용합니다.  
  * **새로운 도메인 추가 시, 에이전트의 핵심 코드를 수정하지 않고 해당 도메인의 플러그인 파일만 생성/수정**하여 재사용성을 보장합니다.

### 2.7.2. 입력/출력 인터페이스 표준화

* **목표:** 에이전트 내의 각 모듈 간 데이터 흐름을 표준화하여 모듈 결합의 유연성을 확보하고 특정 모듈의 교체를 용이하게 합니다.  
* **구현:**  
  * **문헌 정보:** 검색된 문헌의 메타데이터 및 초안 생성 결과는 모든 모듈(2.1, 2.2, 2.3, 2.4)이 공유하는 **표준화된 클래스 구조 또는 JSON Schema**를 따릅니다.  
  * **데이터 형식:** NAS 연동(2.2) 시 다운로드/업로드되는 모든 연구 자료와 결과 보고서(2.5)의 포맷을 표준화하여 호환성을 높입니다.

### 2.7.3. Config 기반 실행 환경 구축

* **목표:** 연구자가 에이전트를 새로운 환경 또는 도메인에서 실행할 때 설정 파일을 통해 손쉽게 환경을 변경할 수 있도록 합니다.  
* **구현:**  
  * 에이전트 실행에 필요한 모든 파라미터(조사 대상 도메인, 기본 검색 키워드, NAS 경로, LLM API 키, 모델 버전 등)를 **단일 YAML 또는 INI Config 파일**로 관리합니다.  
  * **조사 대상을 변경할 경우, 해당 Config 파일만 수정**하여 에이전트를 재시작하며, 코드 수정은 최소화합니다.  
  * 이 Config 파일은 버전 관리 시스템(Git)으로 관리하여 환경 설정 변경 이력을 추적합니다.
